# 文档切块升级记录

## 1. 文档目的

本文档用于记录当前项目在 PDF 解析、文本清洗与 Chunking 方面的实现现状、工程判断以及后续升级方向，便于用于：

- 项目技术记录
- 版本迭代规划
- 毕业设计说明书中的“优化与扩展方向”
- 答辩时说明当前方案与工业级方案的差异

---

## 2. 当前方案概述

当前项目的文档切块链路主要由两个服务完成：

- `PdfParseService`
- `ResumeChunkingService`

当前处理流程如下：

1. 使用 `PDFBox` 按页提取 PDF 文本
2. 对文本进行基础清洗
3. 按双换行进行初步分段
4. 使用启发式规则合并被 PDF 错误拆开的段落
5. 基于关键词和标题行识别 `section`
6. 生成 `PARENT/CHILD` 两层 chunk
7. 后续由 embedding、retrieval、rerank 等模块继续处理

当前方案的核心特点：

- 实现简单，依赖少，容易快速落地
- 不依赖额外模型，成本低，链路稳定
- 规则可解释，便于测试和调试
- 适合作为当前项目阶段的 baseline

---

## 3. 当前方案的能力边界

当前方案可以较好地处理“数字原生、结构相对清晰”的普通简历 PDF，但仍存在明显边界：

- 主要依赖字符串规则与启发式判断
- 不理解 PDF 的真实版面结构
- 对复杂模板、双栏布局、表格简历支持有限
- `section` 识别仍偏粗粒度
- 段落合并主要依赖局部上下文，泛化能力有限
- chunk 粒度仍偏静态，未根据 query 动态调整

换句话说，当前实现属于：

> 一个可运行、可测试、可解释的轻量化 chunking baseline，而不是最终工业级方案。

---

## 4. 工程判断：为什么当前仍采用规则驱动

尽管学术研究中经常强调 Semantic Chunking，即使用 NLP 模型或 LLM 判断语义边界进行切分，但在大厂实际落地中，结构化切分与规则驱动方案通常仍然具有更高的综合性价比。

主要原因如下：

### 4.1 成本更低

规则型切块主要是本地字符串处理，不需要额外模型推理成本，适合高并发与批量入库场景。

### 4.2 延迟更低

相比 CV、NLP 或 LLM 推理，规则处理更快，更适合做基础链路。

### 4.3 稳定性更高

同一输入在规则方案下通常有一致输出，而模型方案存在一定漂移与版本波动。

### 4.4 可解释性更强

规则错了，可以快速定位；模型错了，排查成本通常更高。

### 4.5 易于回归测试

规则型逻辑适合通过单元测试直接锁定行为，不容易在后续改动中悄悄回退。

### 4.6 对结构化文档已经足够有效

很多企业文档天然具有结构，例如：

- Markdown 文档
- 接口文档
- FAQ
- Wiki 页面
- 简历

对于这类文档，优先尊重原始结构，往往比直接做全量语义切分更稳。

因此，更贴近工业实践的表述不是“规则比 AI 高级”，而是：

> 在文档切块这件事上，工程落地更倾向于让规则承担主链路，让 NLP、LLM、CV 作为增强模块，而不是直接替代基础切块链路。

---

## 5. 未来升级的总体思路

未来升级不应简单理解为“继续增加更多 if-else”，而应理解为：

> 将当前的纯文本启发式切块，逐步升级为“结构化解析 + 语义识别 + 分层检索”的组合架构。

更合理的技术演进不是单一路线，而是多层协同：

- 规则层：基础清洗、兜底、约束
- 模型层：复杂语义识别与结构抽取
- 检索层：rerank、补偿与压缩

---

## 6. 主要升级方向

## 6.1 结构化解析升级

### 当前情况

当前只保留：

- 页码
- 页面纯文本

### 问题

丢失了大量版面信息，例如：

- 标题与正文的视觉差异
- 字体大小与粗细
- 双栏阅读顺序
- 页眉页脚位置
- 图标、表格、标签等布局信息

### 升级方向

后续可将 PDF 解析结果升级为结构化对象，例如：

- `page`
- `block`
- `line`
- `span`

并记录：

- 坐标
- 字体大小
- 是否粗体
- 对齐方式
- 块之间的顺序关系

### 预期收益

- 标题识别更稳定
- 阅读顺序恢复更准确
- 页眉页脚更容易过滤
- 为后续 NLP/LLM/CV 模块提供更高质量输入

---

## 6.2 section 识别升级

### 当前情况

当前 `section` 识别主要依赖关键词匹配。

### 问题

- 对标题变体支持有限
- 对英文或中英混排模板鲁棒性不够
- 非标准标题容易识别失败
- 标签字段容易干扰 `section` 判断

### 升级方向

可将 `section` 识别拆成独立模块，例如：

- `SectionHeadingDetector`
- `SectionClassifierService`

先判断是不是标题，再判断属于哪个 `section`。

### 可选实现

- 规则 + 版面特征
- 小型文本分类模型
- LLM 做辅助分类

---

## 6.3 段落修复升级为文档块重建

### 当前情况

当前 `mergeParagraphs()` 主要做相邻段落是否应拼接的判断。

### 问题

这种方式只能修局部断句，不能真正理解简历块结构。

### 升级方向

后续可从“段落修复”升级为“文档块重建”，识别出更完整的结构块，例如：

- 个人信息块
- 教育经历块
- 项目块
- 技能块
- 奖项块

项目块内部还可进一步结构化为：

- 项目名称
- 项目背景
- 技术栈
- 职责
- 成果/指标

### 预期收益

- 检索单元更接近真实语义边界
- 项目级问答效果更好
- 后续生成 STAR 时上下文更完整

---

## 6.4 分层 chunking 升级

### 当前情况

当前已实现 `PARENT/CHILD` 两层 chunk。

### 问题

- 粒度仍偏静态
- 所有 query 共用同一套切块结果

### 升级方向

后续可构建多层 chunk 体系，例如：

- Document-level
- Section-level
- Block-level
- Sentence-level

### 预期收益

不同问题使用不同粒度：

- 总结类问题：section/document 级
- 细节问答：block/sentence 级
- 项目介绍：project block 级

---

## 6.5 query-aware 检索升级

### 当前情况

当前采用静态切块后统一检索。

### 问题

不同 query 的最优证据粒度并不相同。

### 升级方向

后续可根据 query 意图动态选择召回粒度，例如：

- “介绍你的项目经历” -> section / project block
- “QPS 提升了多少” -> 指标句 / 结果句
- “这份简历适合什么岗位” -> summary + skill + work 聚合块

### 预期收益

- 召回更精准
- 冗余更少
- context compression 压力更小

---

## 6.6 文本清洗演化为独立 pipeline

### 当前情况

当前 `cleanText()` 将清洗逻辑集中写在一个方法中。

### 问题

- 随着规则增多，可读性下降
- 某一步效果不好时不易替换
- 测试粒度不够细

### 升级方向

可拆分为多个清洗步骤，例如：

- `CharacterNormalizeStep`
- `PrivateUseCleanupStep`
- `HeaderFooterCleanupStep`
- `BulletCleanupStep`
- `WhitespaceNormalizeStep`
- `LayoutRepairStep`

### 预期收益

- 模块职责更清晰
- 每一步都可独立测试
- 更适合后续灰度或开关控制

---

## 6.7 使用 NLP 做语义分段增强

### 当前情况

当前语义分段主要依赖：

- 标题识别
- 标点
- 换行
- 前后字符启发式

### 升级方向

后续可引入 NLP 模型做：

- 句子边界识别
- 主题边界识别
- section 分类
- 项目块结构抽取

### 更适合的位置

NLP 更适合作为规则层之上的精修模块，例如：

- 判断两段是否应合并
- 判断某段属于哪个 `section`
- 抽取项目名、技术栈、职责、成果等结构化字段

### 工程评价

NLP 可以提升语义理解能力，但通常不应直接替代整个基础切块链路。

---

## 6.8 使用 LLM 做高层语义增强

### 不建议直接用 LLM 替代底层解析的原因

- PDF 原始输入对 LLM 不友好
- 成本高
- 输出不够稳定
- 入库结构难以严格控制
- 调试成本高

### 更合理的使用位置

LLM 更适合放在高层语义阶段，例如：

- section 辅助分类
- 项目块语义重组
- STAR 要素抽取
- 项目摘要生成
- 复杂边界纠偏

### 推荐定位

LLM 应被视为：

> 规则与结构化解析之上的语义增强层，而不是底层解析层的完全替代。

---

## 6.9 使用计算机视觉做版面理解

### 当前情况

当前方案不理解页面视觉布局。

### CV 可解决的问题

- 扫描件 OCR
- 标题检测
- 段落块检测
- 双栏阅读顺序恢复
- 页眉页脚识别
- 图标/标签区域识别

### 适用场景

当 PDF 出现以下情况时，CV 方案更有价值：

- 扫描版 PDF
- 双栏简历
- 表格简历
- 图标字体较多
- 复杂图文混排

### 推荐定位

CV 更适合作为结构化解析层的一部分，用于恢复块边界与阅读顺序，而不是单独直接替代整个 chunking 流程。

---

## 7. 工业级落地的推荐组合

更贴近工业落地的方案通常不是单一技术路线，而是多层协同：

### 7.1 规则层

负责：

- 去乱码
- 去零宽字符
- 去装饰符号
- 去页眉页脚
- 初步标题候选判断

特点：

- 低成本
- 高稳定
- 易测试
- 易解释

### 7.2 模型层

负责：

- section 分类
- block 边界识别
- 项目块重建
- 信息抽取

特点：

- 更懂语义
- 适合处理规则难覆盖的复杂情况

### 7.3 检索层

负责：

- hybrid retrieval
- rerank
- parent-child 回填
- context compression

特点：

- 对前面不完美的 chunk 边界做效果补偿

### 7.4 推荐结论

更合理的工程策略不是“只用规则”或“只用模型”，而是：

> 规则做基础链路，模型做增强，检索做补偿。

---

## 8. 面向当前项目的推荐演进顺序

如果后续继续升级，建议按以下顺序推进：

### 第一阶段：继续增强规则 baseline

- 完善 `section` 标题词典
- 增加标签字段识别
- 增加页眉页脚过滤
- 优化段落合并规则

### 第二阶段：引入结构化解析能力

- 解析页面坐标
- 识别 block / line
- 恢复阅读顺序

### 第三阶段：引入 NLP 模块

- section 分类
- 语义分段
- 项目块重建

### 第四阶段：引入 LLM 增强

- 项目结构抽取
- STAR 要素抽取
- 复杂语义纠偏

### 第五阶段：引入 CV / OCR

- 扫描件支持
- layout detection
- 复杂版式解析

这个顺序的原因是：

- 先提升 baseline，收益最快
- 再增强结构理解
- 最后再引入成本更高的模型能力

## 9. 可直接用于说明书或答辩的正式表述

可参考如下表述：

> 当前系统采用基于规则的轻量化 PDF 文本切块方案，优先保证上传、解析、分块、检索链路的可运行性、低成本与可解释性。该方案已经能够满足一般数字原生简历 PDF 的基础入库与检索需求，但在复杂版式、结构恢复、语义边界识别与动态粒度检索方面仍存在局限。后续可沿结构化解析、section 分类、文档块重建、分层 chunking、query-aware retrieval 以及 NLP/LLM/CV 增强等方向持续演进，使系统逐步从启发式文本切块升级为更接近工业级的文档理解与检索架构。

另一个更强调工程取舍的表述如下：

> 尽管学术研究中广泛讨论 Semantic Chunking，即利用 NLP 模型或大语言模型对文本语义边界进行切分，但在实际工程落地中，结构化切分与规则驱动方案仍然具有更高的综合性价比。其主要原因在于，企业场景下的文档处理不仅关注切分精度，还必须同时考虑系统延迟、推理成本、结果稳定性、可解释性以及可维护性。对于大多数具有天然结构的信息载体，例如 Markdown 文档、知识库页面、接口文档、FAQ 以及简历等，基于标题层级、段落边界、列表结构和版面特征的结构化切分，往往已经能够提供足够高质量的检索单元。在此基础上，再结合 rerank、parent-child chunk、metadata filter 与 context compression 等检索增强手段，通常可以以更低成本达到较好的整体效果。因此，在工业实践中，CV、NLP 与 LLM 更适合作为增强模块，重点用于复杂版式文档、语义边界模糊样本、结构抽取与高价值场景中的精修，而不是直接替代基础切块链路。当前项目采用规则驱动的轻量切块策略，正是基于上述工程权衡：优先保证链路稳定、成本可控与实现可解释，在此基础上为后续引入更高级的语义增强能力预留扩展空间。

---

## 10. 当前结论

当前 chunking 方案不是最终形态，但作为项目现阶段的工程 baseline 是合理的。  
未来真正的升级重点，不是继续堆砌更多局部判断语句，而是逐步建立：

- 更强的结构解析能力
- 更清晰的 section 分类模块
- 更合理的 block 重建能力
- 更灵活的分层和动态检索能力
- 规则、模型、检索协同的整体架构

这部分内容可作为后续版本规划、论文扩展方向以及答辩说明的技术依据。
