# Document Chunking 升级与扩展记录

## 1. 文档目的

本文档用于记录当前项目在“PDF 文档解析与 Chunking”方面的现状、问题边界，以及后续可扩展的升级方向。  
内容主要服务于以下场景：

- 项目阶段性技术总结
- 后续版本迭代规划
- 毕业设计说明书中的“优化方向/扩展方向”
- 答辩时对现有实现与工业级方案差异的说明

---

## 2. 当前实现概述

当前项目的文档切块链路主要包含两个核心服务：

- `PdfParseService`
- `ResumeChunkingService`

### 2.1 当前处理流程

1. 使用 `PDFBox` 从 PDF 中按页提取文本
2. 对提取文本做基础清洗
3. 按双换行做初步分段
4. 使用启发式规则合并被错误拆断的段落
5. 使用关键词规则识别 section
6. 基于标题行做 section 继承
7. 将每个语义段落构造成 `PARENT` chunk
8. 对过长段落再切分为 `CHILD` chunk

### 2.2 当前方案特点

- 实现简单，依赖少，容易快速落地
- 不依赖外部模型，成本低，稳定性较高
- 规则可解释，便于调试和写测试
- 适合当前项目阶段作为 baseline 使用

### 2.3 当前方案局限

- 主要依赖字符串规则与启发式判断
- 不理解 PDF 版面结构
- 对复杂模板、双栏布局、图标字体、表格布局支持有限
- section 识别仍偏粗粒度
- 段落合并主要依赖局部字符上下文，泛化能力有限
- chunk 粒度仍偏静态，未结合 query 动态调整

---

## 3. 当前方案为什么仍然合理

虽然当前方案不属于工业级最终形态，但在当前项目阶段仍然合理，原因如下：

1. 先保证主链路可运行  
当前项目的第一目标是打通“上传 -> 解析 -> 分块 -> 检索 -> 回答”闭环。

2. 工程复杂度可控  
规则型 chunking 不依赖额外训练数据，也不引入复杂模型推理链路。

3. 便于测试和调试  
规则型方案适合快速定位问题，例如乱码、错误 section、断段等。

4. 适合作为后续升级基线  
后续无论引入 NLP、LLM 还是计算机视觉，当前实现都可以作为对照版本。

因此，当前方案更适合定义为：

> 一个可运行、可测试、可解释的轻量化 baseline chunking 方案。

---

## 4. 升级方向总览

未来升级不应简单理解为“把 if-else 写得更长”，而应理解为：

> 将 chunking 从纯文本启发式处理，逐步升级为“结构化解析 + 语义识别 + 分层检索”的组合架构。

本文建议的升级方向主要分为以下几类：

1. 结构化解析升级
2. section 识别升级
3. 段落重建升级
4. 分层 chunking 升级
5. query-aware 检索升级
6. 清洗 pipeline 升级
7. NLP / LLM / CV 能力增强

---

## 5. 升级方向一：从纯文本解析升级到结构化解析

### 5.1 当前方式

当前使用 `PDFBox` 直接抽取纯文本，只保留：

- 页面内容
- 页码

### 5.2 当前问题

纯文本提取会丢失很多文档结构信息，例如：

- 标题与正文的视觉差异
- 段落边界
- 字体大小和粗细
- 双栏阅读顺序
- 页眉页脚位置
- 图标、标签、表格等版面信息

### 5.3 升级方向

后续可将 PDF 文档解析为更丰富的结构化对象，例如：

- page
- block
- line
- span

并记录更多元数据：

- 文本坐标
- 字体大小
- 是否粗体
- 对齐方式
- 块之间的上下关系

### 5.4 预期收益

- 标题识别更稳定
- 页眉页脚和装饰噪声更容易过滤
- 复杂模板简历的阅读顺序恢复更准确
- 为后续 NLP/LLM/CV 模块提供更高质量输入

---

## 6. 升级方向二：section 识别从关键词规则升级为独立能力模块

### 6.1 当前方式

当前 `detectSection()` 主要依赖关键词匹配，例如：

- 项目经历 -> `PROJECT`
- 教育经历 -> `EDUCATION`
- 专业技能 -> `SKILL`

### 6.2 当前问题

- 对标题变体支持有限
- 对英文/中英混排模板鲁棒性不够
- 对非标准标题识别不稳定
- “技术栈：...”这类标签字段容易干扰 section 判断

### 6.3 升级方向

后续可将 section 识别拆成独立模块，例如：

- `SectionHeadingDetector`
- `SectionClassifierService`

先判断“是不是标题”，再判断“属于哪个 section”。

### 6.4 可选技术路径

- 规则 + 版面特征
- 小型文本分类模型
- LLM 做 section tagging

### 6.5 预期收益

- section 判断更清晰
- 模块职责更单一
- 后续更容易替换为更强实现

---

## 7. 升级方向三：从段落修复升级到文档块重建

### 7.1 当前方式

当前 `mergeParagraphs()` 本质上是在做“相邻两段是否需要拼接”的判断。

### 7.2 当前问题

这种方式只能修复局部断句，例如：

- “微” / “服务架构”
- “项目背景：” / “后续说明”

但它并不能真正理解简历的结构块。

### 7.3 升级方向

后续可从“段落修复”升级为“文档块重建”，识别出更完整的结构块，例如：

- 个人信息块
- 教育经历块
- 项目块
- 技能块
- 奖项块

进一步还可对项目块做细分：

- 项目名称
- 项目背景
- 技术栈
- 职责
- 结果

### 7.4 预期收益

- 检索单元更接近真实语义边界
- 项目级问答效果更好
- 后续生成 STAR、面试回答时上下文更完整

---

## 8. 升级方向四：从固定切块升级为分层 chunking

### 8.1 当前方式

当前实现已经具备 `PARENT/CHILD` 两层 chunk。

### 8.2 当前问题

虽然比单层切块更好，但仍然偏静态：

- 所有问题共用同一套切块结果
- 无法根据问题粒度调整召回对象

### 8.3 升级方向

后续可构建多层 chunk 体系，例如：

- Document-level chunk
- Section-level chunk
- Block-level chunk
- Sentence-level chunk

### 8.4 预期收益

不同类型问题可匹配不同粒度：

- 总结型问题 -> section/document 级
- 细节问答 -> block/sentence 级
- 项目解释 -> project block 级

---

## 9. 升级方向五：引入 query-aware retrieval 或动态粒度召回

### 9.1 当前方式

当前系统使用静态 chunk 入库，再统一检索。

### 9.2 当前问题

不同 query 的最优证据粒度不同，静态 chunk 不一定适合所有问题。

### 9.3 升级方向

根据 query 意图动态选择召回粒度，例如：

- “介绍你的项目经历” -> section / project block
- “QPS 提升了多少” -> 结果句 / 指标句
- “这份简历适合什么岗位” -> summary + skill + work 聚合块

### 9.4 预期收益

- 检索更精准
- 冗余更少
- context compression 压力更小

---

## 10. 升级方向六：将文本清洗演化为独立 pipeline

### 10.1 当前方式

当前 `cleanText()` 将清洗逻辑集中写在一个方法内。

### 10.2 当前问题

- 扩展步骤多了之后可读性会下降
- 某一步效果不好时不容易单独替换或关闭
- 测试粒度不够细

### 10.3 升级方向

后续可拆为 pipeline，例如：

- `CharacterNormalizeStep`
- `PrivateUseCleanupStep`
- `HeaderFooterCleanupStep`
- `BulletCleanupStep`
- `WhitespaceNormalizeStep`
- `LayoutRepairStep`

### 10.4 预期收益

- 模块职责更清晰
- 每一步都可独立测试
- 后续可以灰度开关某些清洗规则

---

## 11. 升级方向七：使用 NLP 模型做语义分段

### 11.1 当前方式

当前语义分段主要依赖：

- 标题判断
- 标点和换行
- 前后字符启发式

### 11.2 升级方向

后续可引入 NLP 模型处理：

- 句子边界识别
- 主题边界识别
- section 分类
- 项目块结构抽取

### 11.3 更适合的使用位置

NLP 更适合做“规则之上的精修”：

- 判断两段是否属于同一语义单元
- 判断一段属于哪个 section
- 把项目名、技术栈、职责、结果抽成结构化字段

### 11.4 预期收益

- 比纯规则更懂语义
- 对不同简历模板泛化更好
- 复杂断句与非标准结构处理更稳定

### 11.5 注意事项

- 需要训练数据或模型支持
- 部署与维护成本更高
- 工程复杂度高于规则方案

---

## 12. 升级方向八：使用 LLM 做高层语义增强，而不是直接替代底层解析

### 12.1 为什么不建议直接用 LLM 做全量 PDF 解析

- PDF 原始输入对 LLM 不友好
- 成本高
- 输出不够稳定
- 入库结构难以严格控制
- 调试成本高

### 12.2 更合理的方式

LLM 更适合放在高层语义阶段，例如：

- section 辅助分类
- 项目块语义重组
- STAR 信息抽取
- 项目摘要生成
- 复杂段落合并判断

### 12.3 推荐定位

应将 LLM 定位为：

> 规则和结构化解析之上的语义增强层，而不是底层解析层的完全替代。

---

## 13. 升级方向九：使用计算机视觉做版面理解

### 13.1 当前方式

当前只做文本解析，不理解页面视觉布局。

### 13.2 计算机视觉可解决的问题

- 扫描件 OCR
- 标题检测
- 段落块检测
- 双栏阅读顺序恢复
- 页眉页脚识别
- 图标/标签区识别

### 13.3 适用场景

当 PDF 出现以下情况时，CV 方案更有价值：

- 扫描版 PDF
- 双栏简历
- 复杂排版
- 表格简历
- 图标字体较多

### 13.4 推荐定位

计算机视觉更适合作为：

> 结构化解析层的一部分，为后续 chunking 提供更可靠的文档块边界与阅读顺序。

---

## 14. 更接近工业级的组合方案

工业级方案通常不是单一技术路线，而是多层协同：

### 14.1 规则层

负责确定性、低成本、强约束的处理：

- 去乱码
- 去零宽字符
- 去页眉页脚
- 去装饰符号
- 基础标题候选判断

### 14.2 模型层

负责复杂语义判断：

- section 分类
- block 边界识别
- 项目块重建
- 信息抽取

### 14.3 检索层

负责最后的效果修正：

- hybrid retrieval
- rerank
- parent-child 回填
- context compression

### 14.4 总体思路

更合理的工程策略不是“完全依赖规则”或“完全依赖模型”，而是：

> 规则做基础清洗和兜底，模型做语义增强，检索层做最终补偿。

---

## 15. 面向当前项目的推荐升级顺序

如果后续真的要继续升级，建议按如下顺序推进，而不是一次性推倒重写：

### 第一阶段：继续增强规则 baseline

- 完善 section 标题词典
- 增加更多标签字段识别
- 增加页眉页脚噪声过滤
- 优化段落合并规则

### 第二阶段：引入结构化解析能力

- 解析页面坐标
- 识别 block / line
- 恢复阅读顺序

### 第三阶段：引入 NLP 模块

- section 分类
- 语义分段
- 项目块重建

### 第四阶段：引入 LLM 做增强

- 项目结构抽取
- STAR 要素抽取
- 复杂语义纠偏

### 第五阶段：引入视觉版面理解

- OCR
- layout detection
- 扫描件支持

---

## 16. 可用于项目说明书/答辩的表述

可直接参考如下表述：

> 当前系统采用基于规则的轻量化 PDF 文本分块方案，优先保证上传、解析、分块、检索链路的可运行性、低成本与可解释性。该方案已能够满足一般数字原生简历 PDF 的基础入库与检索需求，但在复杂版式、结构恢复、语义边界识别与动态粒度检索方面仍存在局限。后续可沿结构化解析、section 分类、文档块重建、分层 chunking、query-aware retrieval，以及 NLP/LLM/CV 增强等方向持续演进，使系统逐步从启发式文本切块升级为更接近工业级的文档理解与检索架构。 

---

## 17. 当前结论

当前 chunking 方案不是最终形态，但作为项目现阶段的工程 baseline 是合理的。  
未来真正的升级重点不是继续堆砌更多局部判断语句，而是逐步建立：

- 更强的结构解析能力
- 更清晰的 section 分类模块
- 更合理的 block 重建能力
- 更灵活的分层与动态检索能力
- 规则、模型、检索协同的整体架构

这部分内容可作为后续版本规划与论文扩展方向的技术依据。
